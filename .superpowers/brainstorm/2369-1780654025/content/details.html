关键细节设计决策

以下 5 个细节决策将直接影响实现方案

1. 阅读器的核心交互

A

连续滚动模式

所有章节内容加载为连续滚动，类似网页小说。自动记录阅读位置。优点：流畅自然，适合长章节。缺点：大章节内存占用高。

B

分页模式

按章分页，每章独立页面，左右切换或上下翻页。优点：加载快，内存可控，进度精确。缺点：切换有顿挫感。

C

虚拟滚动 + 懒加载

当前章节在视口内完整渲染，相邻章节预加载，滚动时自动切换。优点：兼顾流畅和性能。缺点：实现复杂度高。

2. 首页/书架的内容组织

D

书架即首页

登录后直接看到自己的书架（收藏+最近阅读），推荐内容嵌入书架底部。零导航成本，打开即读。

E

发现流 + 书架Tab

首页是推荐/排行/分类的发现流，书架是独立 Tab。适合喜欢探索新书的用户。

3. 管理端的数据操作模式

F

表格 + 弹窗编辑

列表页展示数据表格，点击行弹出编辑表单。操作集中，不离开当前页。适合批量处理。

G

独立详情页

列表点击跳转到独立编辑页，完整表单。适合复杂编辑场景，表单字段多。

H

内联编辑

表格单元格直接可编辑，修改后自动保存。类似 Airtable/Notion 数据库。最高效。

4. 认证与权限体系

I

JWT + RBAC（推荐）

保留 JWT 认证，增加基于角色的访问控制（reader/user/admin）。Token 中包含角色，API 按角色鉴权。

J

Session + CSRF

改用传统 Session 认证，HttpOnly Cookie + CSRF Token。更安全，但跨端支持弱。

5. 技术栈选择

保持现有栈

React + Vite + Tailwind + Zustand + React Query
成熟稳定，团队熟悉，迁移成本低

升级 Next.js

Next.js 15 App Router + Server Components + Tailwind
SSR 首屏快，SEO 好，适合内容型应用
